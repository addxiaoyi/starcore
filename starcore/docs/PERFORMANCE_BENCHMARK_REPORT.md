## 性能瓶颈分析

### 可能的瓶颈点

1. **数据库连接开销**
   - 每次操作都需要获取和释放连接
   - 优化建议: 使用连接池（已实现 HikariCP）

2. **序列化/反序列化开销**
   - Properties ↔ SQL 的转换过程
   - 优化建议: 缓存热点数据

3. **磁盘 I/O**
   - SQLite 的磁盘写入特性
   - 优化建议: 使用 WAL 模式、调整同步策略

4. **查询效率**
   - 表扫描 vs 索引查询
   - 优化建议: 添加必要的索引

## 优化建议

### 短期优化（已实现）

1. ✅ 使用 HikariCP 连接池
2. ✅ 实现异步保存
3. ✅ SQL 表结构优化
4. ✅ 批量操作支持

### 中期优化（可选）

1. **添加缓存层**
   ```java
   // 使用 Caffeine 缓存热点数据
   Cache<String, Properties> cache = Caffeine.newBuilder()
       .maximumSize(100)
       .expireAfterWrite(5, TimeUnit.MINUTES)
       .build();
   ```

2. **SQLite 性能调优**
   ```sql
   PRAGMA journal_mode = WAL;
   PRAGMA synchronous = NORMAL;
   PRAGMA cache_size = 10000;
   PRAGMA temp_store = MEMORY;
   ```

3. **批处理优化**
   - 合并多个小事务为大事务
   - 使用批量 INSERT/UPDATE

### 长期优化（高级）

1. **迁移到 MySQL/PostgreSQL**
   - 更好的并发性能
   - 更强大的查询优化器
   - 更好的事务隔离

2. **分布式缓存**
   - Redis 作为热数据缓存
   - 减少数据库访问频率

3. **读写分离**
   - 主从复制架构
   - 读操作分散到从库

### B. 测试数据说明

基准测试使用 1000 个键值对作为测试数据，每个键值对格式如下:

```
test.key.0 = test.value.0.data.UUID
test.key.1 = test.value.1.data.UUID
...
test.key.999 = test.value.999.data.UUID
```

这模拟了真实的国家数据存储场景。

解决方案:

解决方案:
- 关闭其他占用资源的程序
- 多次运行取平均值
